# Findings & Decisions

## Requirements
- Build documentation for Pin2pin.ai's first tool, focused on a datasheet deep-reading MVP for electronics engineers.
- Capture prior discussion in persistent files inside the project workspace.
- Keep PRD focused on product requirements, workflows, value, and user journeys.
- Separate product design, interaction, and technical architecture from the PRD.
- Reflect key product constraints:
- Single project corresponds to a specific MPN.
- The tool targets Chinese-speaking electronics engineers.
- The main outputs are a simplified chip analysis report page and a CSV parameter export.
- Parameter fields vary by material category.
- Important results should link back to original PDF evidence.
- PDF viewing and exported/previewed artifacts should support branding such as corner marks and watermarks.

## Technical Decisions
| Decision | Rationale |
|----------|-----------|
| Use one project per specific MPN | Matches how engineers actually work and evaluate chips |
| Report page is a simplified view, not a full parameter table | Better for reading, understanding, and sharing |
| CSV export is a separate artifact optimized for copy/paste reuse | Engineers commonly move data into other tools and workflows |
| Parameter outputs must be category-aware | Different chip types require different key fields |
| Evidence traceability is a first-class requirement | Trust and validation are central to the product value |
| Keep defensive constraints explicit in docs | Large files, low-quality PDFs, and abusive usage can distort MVP cost and UX |
| The first commercial wedge should be replacement evaluation, not a generic hardware copilot | The interviews repeatedly converge on this as the sharpest, most urgent, and most explainable value proposition |
| Manual delivery should validate payment and repeat usage before heavy build-out | Pain is validated more strongly than monetization, so the next step should close that gap |
| PRD detail should be organized around the wedge workflow, not broad capability lists | This keeps future implementation decisions anchored to one concrete user task |
| OpenDataLoader should be the primary parsing foundation for the rapid MVP | It maps well to datasheet structure extraction and evidence-linked UI needs without requiring custom parsing from scratch |
| A handoff doc should be maintained once the repo becomes doc-heavy | The project now spans multiple files and needs a clean restart path |
| The first product UI should follow a chat-first model similar to the sister product rather than a dashboard workbench | The user explicitly wants an AI-native interaction where the conversation remains the main stage |
| The first coded slice should stop at a usable single-PDF closure | This keeps implementation narrow enough to verify interaction quality before parser/back-end integration |
| The first prototype should use deterministic seeded analysis data | It enables a testable UI and export loop without blocking on parser installation and infrastructure |
| The next MVP slice should replace client-only seed analysis with a narrow server-side real-analysis path | This closes the highest-value gap without reopening multi-PDF comparison scope |
| The real-analysis contract should expose `complete`, `partial`, and `failed` explicitly | The chat-first UI needs honest trust signals instead of pretending every upload is equally parseable |
| The parser stack should prefer a real library before system tools and raw fallbacks | This makes the local MVP more portable and materially improves extraction quality without changing the UI contract |
| The UI should use the same async job shape that the future production parser service will use | This lets the MVP evolve from local in-memory jobs to a real queue without rewriting the chat-first workflow |
| The main repo docs now need a deliberate sync pass from the shipped code | Without that sync, future sessions will keep starting from stale wedge/scope assumptions and duplicate already-finished work |

## Issues Encountered
| Issue | Resolution |
|-------|------------|
| The project directory had no existing documentation structure | Create docs from scratch with a clean separation by purpose |
| Initial visual direction drifted away from the provided chat-first reference | Reworked the interaction model before coding and implemented the prototype around the corrected direction |
| Local environment did not have PDF parsing libraries or CLI tools preinstalled | Keep the first prototype local and deterministic, and leave real parsing for the next integration phase |
| A true production parser is still not installed in the repo | Implement a minimal local server-side parser first so the UI can leave pure filename-seed mode without waiting on infrastructure |
| The repository root is not a Git repository | Use file timestamps, planning files, and direct source inspection for state review until version control is initialized here |

## Combined Review Synthesis
- The current single-PDF real-analysis MVP is architecturally acceptable for internal or small-scale testing, but it is not yet strong enough for broader external use.
- The core problem is credibility, not scope:
- engineering gaps weaken truthfulness, durability, and parser trust
- design gaps weaken hierarchy, confidence, and perceived product intent
- These two review tracks converge on the same conclusion:
- do not expand product scope yet
- spend the next implementation wave on trust, clarity, and recovery quality

## Combined Work Checklist
1. Upload guardrails and expectation-setting
- Reject unsupported or clearly low-quality uploads earlier.
- Explain analysis limits and likely degraded cases before the job starts.
- Use the empty right panel to teach the evidence-validation workflow instead of leaving it visually blank.
2. Honest grounded follow-up behavior
- Hide or disable the follow-up composer until the first analysis result is available.
- Ensure follow-up answers are grounded in extracted evidence, and explicitly say when the system does not know.
- Unify the core workflow copy to Chinese-first product language.
3. Durable state and recovery
- Replace the in-memory job store in `lib/analysis-jobs.ts` with persistent state.
- Make refresh/reopen behavior reliable enough that users can trust longer parsing sessions.
4. Parser and evidence hardening
- Improve extraction reliability in `lib/server-analysis.ts`.
- Reduce heuristic evidence placement and make `complete` / `partial` differences user-visible and meaningful.
5. Visual hierarchy and trust polish
- Give the empty state a stronger product narrative and clearer primary action.
- Strengthen hierarchy across task, summary, review, parameters, and evidence sections in `components/workspace.tsx`.
- Demote or remove low-value first-run stats until they represent real signal.

## Hardening Wave 1 Outcomes
- Upload guardrails are now shared between UI and API through `lib/upload-validation.ts`.
- Wrong-type, empty, and oversized uploads now fail before long parsing begins.
- Analysis job snapshots are now written to local disk through `lib/analysis-store.ts` instead of living only in memory.
- The follow-up composer is now hidden until a first result exists.
- Follow-up answers now state an explicit boundary: they are grounded only in the extracted first-pass result.
- Parser/evidence quality and timeout UX are now the clearest remaining MVP hardening gaps.
